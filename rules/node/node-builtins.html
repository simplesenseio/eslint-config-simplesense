# node/no-unsupported-features/node-builtins

disallow unsupported Node.js built-in APIs on the specified version

  • ‚≠êÔ∏è This rule is included in plugin:node/recommended preset.

Node.js community is improving built-in APIs continuously. You can check Node.js Documentation (opens new window) to know which Node.js version supports each Node.js API.

This rule reports unsupported Node.js built-in APIs on the configured Node.js version as lint errors. Editor integrations of ESLint would be useful to know it in real-time.

# üìñ Rule Details

This rule reports APIs of Node.js built-in APIs on the basis of Node.js v13.2.0 Documentation (opens new window).

# Configured Node.js version range

This rule reads the engines (opens new window) field of package.json to detect which Node.js versions your module is supporting.

I recommend the use of the engines (opens new window) field because it's the official way that indicates which Node.js versions your module is supporting. For example of package.json:

{
    "name": "your-module",
    "version": "1.0.0",
    "engines": {
        "node": ">=8.0.0"
    }
}

If you omit the engines (opens new window) field, this rule chooses >=8.0.0 as the configured Node.js version since 8 is the minimum version the community is maintaining (see also Node.js Release Working Group (opens new window)).

# Options

{
    "node/no-unsupported-features/node-builtins": ["error", {
        "version": ">=8.0.0",
        "ignores": []
    }]
}

# version

As mentioned above, this rule reads the engines (opens new window) field of package.json. But, you can overwrite the version by version option.

The version option accepts the valid version range of node-semver (opens new window).

# Known limitations

This rule cannot find non-static things. For example:

  • New properties and methods of instances.
  • New parameters of functions.
  • New options properties of function parameters.
  • New events.
